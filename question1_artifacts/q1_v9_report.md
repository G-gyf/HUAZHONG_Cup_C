# 问题一第九版方案实施报告

## 1. 方案背景

第九版方案的核心目标，是把问题一重新拉回题目原始主目标：**总配送成本最小**。

前几版迭代中，求解过程逐步把注意力放到了路线结构上，例如：

- 路线数是否继续下降
- 单点路线是否继续减少
- mixed-big 路线是否被进一步清理

这些指标有解释价值，但并不是题目真正的主目标。结合旧基线结果可以看到：

- 旧基线总成本为 `92442.10`
- 其中 `startup_cost = 52800`，约占总成本的 `57%`
- `wait + late` 仅约 `873.6`，不到总成本的 `1%`

这说明路线结构确实重要，但是否接受一个解，最终仍应由 `total_cost` 决定，而不是由结构是否“更漂亮”决定。

因此，第九版方案的根本调整是：

- 把 `total_cost` 恢复为唯一主目标
- 把路线结构相关指标全部降级为诊断指标
- 允许有成本价值的 mixed-big 存在
- 用明确的止损机制结束无限迭代

## 2. 第九版方案的总体思路

第九版不是对旧流程的小修小补，而是一次全局重置。整体思想可以概括为：

1. 主优化阶段只看总成本是否下降
2. 路线结构指标继续记录，但只用于解释，不再主导接受逻辑
3. route-pool 列生成改成按真实节约值排序，而不是按结构偏好排序
4. mixed-big 只分类，不再机械清零
5. 固定两轮 cost-first 全局重优化，之后设置止损点
6. 若仍无法进一步降本，再做一次 split/packing 敏感性试验

## 3. 具体实施步骤与优化原因

### 3.1 重置总评价口径：`total_cost` 回到唯一主目标

首先对全局解的评价规则进行了重构。

旧逻辑的问题在于，结构指标在求解过程中被不断抬高，逐渐从“辅助观察项”变成了“实际判优项”。这会导致一个风险：明明某个新解成本更低，却因为结构不够整齐而被拒绝。

因此，第九版明确把求解与选解口径统一为：

- 第一目标：`total_cost` 最小
- 只有在总成本完全相同或极接近时，才使用以下指标做 tie-break：
  - `split_customer_count - mandatory_split_customer_count`
  - `route_count`
  - `single_stop_route_count`

这一步的优化原因是：**让搜索方向重新与题目目标一致**。

### 3.2 把结构指标降级为诊断项，而非主搜索驱动项

在第九版中，下列指标继续保留，但不再控制主优化接受逻辑：

- `single_stop_route_count`
- `routes_with_flexible_units_on_big`
- `bad_big_flexible_route_count`
- `promotion_like_count`
- `piggyback_big_count`
- `blocking_big_flexible_count`

此外，时间相关指标也继续输出：

- `late_positive_stops`
- `max_late_min`
- `after_hours_return_count`

这些指标的作用被重新定义为：

- 帮助解释为什么这个解成本高或低
- 帮助判断某些路线结构是否异常
- 帮助下一轮候选生成

但它们不再直接决定最终主解。

优化原因在于：**结构指标适合做解释，不适合替代题目主目标**。

### 3.3 重构全局 MILP：从“结构纯化主线”改为“成本优先主线”

第九版保留了 route-pool + set partitioning 的整体框架，但把其作用一分为二：

- 主链：cost-first global reoptimization
- 支链：structure diagnostics

主链的作用只有一个：最小化 `total_cost`。主链保留的关键约束包括：

- 每个 `ServiceUnit` 恰好覆盖一次
- 车型库存不超限
- 当前必要拆分客户集合不回退

诊断支线则继续输出结构统计信息，供结果分析使用。

之所以这样改，是因为旧流程中的 `unlock_bad_big_flexible -> compress_*` 更像是在“整理结构”，而不是在坚定地做成本主目标优化。第九版通过重构 MILP 主线，避免搜索再次被结构目标带偏。

### 3.4 重做 route-pool 候选生成：按真实节约值选列

候选列生成是第九版的关键改动之一。

旧思路更偏向于优先挑“更像好结构”的路线。第九版改成统一按真实节约值排序：

`saving = current_cover_cost - candidate_route_cost`

也就是说，候选是否值得进入 route-pool，先看它相对当前覆盖方案能不能真降成本，然后才参考：

- 时间窗重叠程度
- 空间邻近程度
- 路线长度

本轮重点构造了四类候选家族：

- `flex_small`
- `piggyback_big`
- `promotion_like_big`
- `seed/current`

这一步的优化原因是：**route-pool 的列质量必须由成本价值决定，而不是由结构偏好决定**。

### 3.5 mixed-big 语义重定义：允许有成本价值的搭载存在

前几版实践已经暴露出一个问题：把所有 mixed-big 都视为坏结构并强行清零，会让总成本优化被带偏。

因此，第九版把 mixed-big 拆成三类：

- `promotion_like_big`
- `piggyback_big`
- `blocking_big_flexible`

在新逻辑下：

- `promotion_like_big` 可以保留
- `piggyback_big` 可以保留
- `blocking_big_flexible` 只继续统计，不再一律强行清零

第九版强调的原则是：

**只要某条 mixed-big 路线能够降低总成本，并且不破坏当前建模口径，就允许存在。**

优化原因在于：**不是所有 mixed-big 都是坏结构，真正需要警惕的是“阻塞型混载”，而不是所有混载本身。**

### 3.6 设置明确止损点：固定两轮 cost-first route-pool iteration

第九版明确规定，本轮主优化只做两轮 cost-first route-pool iteration：

1. 第 1 轮：求当前成本优先 MILP
2. 基于当前昂贵或碎片化 cluster 生成新列
3. 第 2 轮：重新求 MILP

如果两轮之后仍不能把总成本压低到旧基线之下，则不再继续无边界追逐结构指标，而是转去做一次 split/packing 端的敏感性试验。

这一设计的优化原因是：**第九版必须给整个迭代过程设置停表机制，防止继续陷入结构优化的局部循环。**

### 3.7 增加 split/packing 敏感性试验作为 fallback

第九版没有把 split/packing 试验放进主流程，而是把它设置为 fallback。

触发条件是：

- 两轮 cost-first route-pool iteration 都无法进一步压低总成本

只有在这种情况下，才怀疑当前大车依赖是不是由拆分口径、装载策略或 packing 端的刚性造成的。

这样做的优化原因是：**先把主目标做对，再去怀疑模型边界，而不是反过来。**

### 3.8 完善结果输出与可视化表达

第九版还补齐了结果输出和图形表达层，用于解释“为什么成本下降了”。

具体补充包括：

- 更新 `q1_cost_summary.json`
- 更新 `q1_run_report.md`
- 新增 `q1_route_pool_summary.csv`
- 新增 `plot_question1_results.py`
- 生成 13 张结果图，包括成本构成、车辆结构、路线分布、配送网络图等

这一步的原因是：第九版接受了“成本下降但结构未必更漂亮”的结果，因此必须让最终报告具备充分解释力。

## 4. 第九版方案的验证标准

第九版改变了“什么叫成功”的定义。

成功标准不再是：

- 路线数是否减少
- mixed-big 是否继续下降
- 单点路线是否继续下降

而是改为：

- `split_customer_count = mandatory_split_customer_count`
- 普通客户仍保持单次访问口径
- 新全局解的 `total_cost` 低于旧基线 `92442.0983`

同时，以下指标继续记录，但只作解释，不再作主判优：

- `route_count`
- `single_stop_route_count`
- `vehicle_type_usage`
- `promotion_like_count`
- `piggyback_big_count`
- `blocking_big_flexible_count`
- `late_positive_stops`
- `max_late_min`
- `after_hours_return_count`

## 5. 本轮最终结果

第九版最终得到的主解，相比旧基线结果如下：

- 旧基线总成本：`92442.10`
- 第九版总成本：`87616.61`
- 下降幅度：`4825.49`
- 相对降幅：约 `5.22%`

同时，关键约束口径保持稳定：

- `split_customer_count = 37`
- `mandatory_split_customer_count = 37`
- 未引入额外拆分客户

结构指标方面：

- `route_count = 132`，与旧基线持平
- `single_stop_route_count: 101 -> 97`

这一结果非常重要，因为它说明：

**第九版的成本下降不是靠减少路线数实现的，而是在路线总数不变的情况下，靠更优的路线组合、车型匹配和有成本价值的搭载方式实现的。**

## 6. 大车结构的变化与解释

大车使用并没有被简单压缩掉，而是被重新组织。

最终结果显示：

- 大车路线共 `70` 条
- 其中大车单点路线 `41` 条
- 最终 mixed-big 共 `29` 条

其中更细的结构变化是：

- `piggyback_big: 12 -> 20`
- `promotion_like_big: 5 -> 5`
- `blocking_big_flexible: 10 -> 4`

这组结果说明：

1. mixed-big 并没有被彻底消灭
2. 但真正需要警惕的“阻塞型混载”明显减少了
3. 更多 mixed-big 被转化成了有成本价值的顺带搭载

因此，第九版对大车的态度不是“越少越好”，而是“只保留有成本价值的大车使用方式”。

## 7. 第九版方案为什么是有效的

第九版之所以有效，不是因为它碰巧找到了一组更好的参数，而是因为它做对了三件根本性的事：

1. 把题目主目标重新拉回 `total_cost`
2. 把结构指标从主判优逻辑中降级为解释变量
3. 把候选生成和全局重优化都改成真正的成本优先逻辑

最终得到的结果证明：

- 成本确实下降了
- 约束口径没有被破坏
- mixed-big 没有被机械清零，但坏混载显著减少
- 这轮优化是真正符合题目目标的

## 8. 结论

第九版方案可以概括为：

> 在不增加额外拆分的前提下，通过成本优先的全局重优化，把问题一总配送成本从 `92442.10` 压到 `87616.61`；改进主要来自更优的路线组合、车辆匹配以及有成本价值的 mixed-big 搭载，而不是单纯减少路线数。

因此，这一版可以作为当前建模与算法口径下的问题一最终候选解。

需要注意的是，这一结果并不宣称已经证明全局最优。更准确的结论应当是：

- 它已经明显优于旧基线
- 它是当前模型口径下足够可靠、可交付的强解
- 若后续继续增加搜索预算，预期更可能得到小幅改进，而不是再发生大幅结构性变化

## 9. 后续建议

如果后续还要继续计算，更合理的方向是：

1. 适度增加搜索预算，做一轮 5 至 6 小时级别的重搜索，验证是否还能进一步压低 `total_cost`
2. 若成本已基本收敛，则接受 `87616.61` 作为当前模型口径下的问题一最终答案
3. 若要进一步减少大车或改善时间表现，应改模型惩罚结构，而不是仅依赖更长时间搜索

从当前结果看，第九版已经完成了“回到题目主目标、形成稳定强解、建立可解释结果体系”这三个关键任务。
